### 维度一：输入动力学（键盘与鼠标的物理痕迹）
*采集层：`onDidChangeTextDocument`（增量变化）、`onDidChangeCursorSelection`（光标）*

| 序号 | 指标名称 | 计算公式 / 采集方式 | 独立开发者洞察（为什么看它？） |
| :--- | :--- | :--- | :--- |
| 1 | **净代码增量** | `新增字符数 - 删除字符数`（按语言分别统计） | 传统插件只算“总行数”是耍流氓。**净增量才是你真正留给项目的资产**，负值说明你在做有价值的重构。 |
| 2 | **删除/新增比（DNR）** | 总删除字符数 / 总新增字符数 | **代码洁癖指数**。高于0.5说明你在精雕细琢，低于0.2说明你在疯狂堆屎山。 |
| 3 | **心流击键频率** | 连续击键间隔 < 500ms 的持续时间段 | 剔除思考停顿和摸鱼时间，**算出你真正的“有效编码肌肉耐力”**。 |
| 4 | **撤销深度（峰值）** | 单次连续撤销操作的最大步数（Ctrl+Z连续次数） | **焦虑温度计**。当你一次撤销超过15步，说明当前方案完全跑偏，应立即停下来重新设计。 |
| 5 | **高频重复命令** | 统计 `Ctrl+C/V`、`Alt+↑/↓` 等快捷键的调用密度 | 检测“CV工程师”状态。某天CV率异常飙升 → 你在搬砖而非创造。 |
| 6 | **行内编辑专注度** | 单次未切换文件的情况下，连续编辑同一行的最大次数 | 如果一行代码让你来回改超过10次，说明这行逻辑异常复杂，需要拆分。 |


### 维度二：空间导航与注意力管理（文件与项目游走）
*采集层：`onDidChangeActiveTextEditor`、`onDidOpenTextDocument`、`onDidCloseTextDocument`*

| 序号 | 指标名称 | 计算公式 / 采集方式 | 独立开发者洞察（为什么看它？） |
| :--- | :--- | :--- | :--- |
| 7 | **Tab狂躁指数** | 单位时间（如5分钟内）打开/关闭的文件Tab总数 | **注意力碎片化程度**。指数越高，说明你正在巨大的代码迷宫中找不着北。 |
| 8 | **文件留存潜伏期** | 文件打开到**首次产生有效编辑**的时间差 | 打开文件迟迟不动笔 → 你在回忆上下文。帮你识别哪些项目切换成本最高。 |
| 9 | **项目跳跃成本** | 在Workspace不同根目录间切换时，恢复编码速度所需的时长 | 独立开发者多项目并行。这个数据告诉你**同时维护几个项目是极限**。 |
| 10 | **读/写模式占比** | 滚动/光标移动（读代码） vs 字符输入（写代码）的时间比值 | 独立开发者经常花大量时间读自己的老代码。**读占比 > 70%说明你正在维护遗留系统**，而非开发新功能。 |
| 11 | **代码搜索依赖度** | 调用全局搜索（Ctrl+Shift+F）或符号搜索（Ctrl+T）的频率 | 高频搜索说明你对项目结构不熟悉，或者项目模块划分不够清晰。 |


### 维度三：代码结构熵变（语法树与逻辑复杂度）
*采集层：保存时触发一次轻量级AST解析（需在子进程，避免卡顿）*

| 序号 | 指标名称 | 计算公式 / 采集方式 | 独立开发者洞察（为什么看它？） |
| :--- | :--- | :--- | :--- |
| 12 | **函数圈复杂度增量** | 保存时计算当前函数的 `if/for/while/&&/||` 嵌套得分 | **看着曲线图爬升会产生生理性厌恶**，倒逼你主动拆分函数。这是代码洁癖的自虐神器。 |
| 13 | **刺头函数识别** | 统计单个函数在整个生命周期中被修改的总次数 | 修改次数最多的那个函数，就是**技术债务最集中的“垃圾堆”**，下次重构优先处理。 |
| 14 | **依赖膨胀率** | 统计 `import/require` 语句随时间的净增长曲线 | 一周内新增超过10个NPM包 → 项目正在急速腐烂，警惕“依赖地狱”。 |
| 15 | **注释存活率** | 统计注释被删除的行数（非代码） | 删除过时注释比写新注释更重要。**反映你对文档的真实维护态度**。 |
| 16 | **文件碎片化指数** | 单个文件内，`export` 或 `class` 定义的平均数量 | 如果一个文件导出了超过5个东西（工具函数大杂烩），说明文件职责划分不清。 |


### 维度四：工具链与运行时压力（终端及调试）
*采集层：`onDidCreateTerminal`、`onDidWriteTerminalData`（匹配error/success）、调试会话API*

| 序号 | 指标名称 | 计算公式 / 采集方式 | 独立开发者洞察（为什么看它？） |
| :--- | :--- | :--- | :--- |
| 17 | **构建/测试失败率** | 终端输出中包含 `error` / `failed` 的占比（按天统计） | **今日运势风向标**。失败率 > 30%，今天就不适合发版，只适合写文档。 |
| 18 | **调试器等待时长** | 断点激活后，停留在断点处的总闲置时间 | 停在断点看变量超过2分钟 → 你在做高负荷逻辑推理，**此时极其厌恶被打扰**。 |
| 19 | **终端命令高频词** | 统计集成终端输入的前5个命令（如 `npm run dev`、`git status`） | 如果你高频输入 `console.log` 或 `print`，说明你在依赖“穷人调试法”；高频输入 `git commit` 说明节奏良好。 |
| 20 | **环境切换成本** | 在“写代码”和“跑脚本/查日志”之间切换的频次 | 频繁切换说明你的项目热重载太慢，或者你需要优化自动化脚本。 |


### 维度五：开发者生理节律与疲劳曲线
*采集层：给上述所有原始数据打上统一时间戳（UTC+8本地化）后计算而来*

| 序号 | 指标名称 | 计算公式 / 采集方式 | 独立开发者洞察（为什么看它？） |
| :--- | :--- | :--- | :--- |
| 21 | **黄金高峰时段** | 自动找出每周/每天**错误率最低、净增量最高**的连续2小时 | 知道了大脑的“主频巅峰时刻”，**把最难的核心算法挪到这个时间段写**。 |
| 22 | **疲劳拐点预警** | 连续编码下，当 `删除/新增比（DNR）` 突然飙升超过阈值时触发 | 疲劳时你会疯狂写了删、删了写。**插件在拐点前10分钟提醒休息，能帮你规避大量Bug**。 |
| 23 | **语言切换适应期** | 从语言A切换到语言B后，恢复到平均击键速度所需的分钟数 | 如果你发现从Python切C++需要15分钟找回手感，那就把同语言的任务批量处理。 |
| 24 | **有效编码时长（非开机时长）** | 剔除发呆（击键间隔 > 3分钟）后的纯编辑时间 | 独立开发者容易自我感动（“我在电脑前坐了12小时”）。**只看这个数据**。 |


### 维度六：AI共生与协作深度（大模型介入）
*采集层：基于输入速度启发式（<100ms连续插入多行）推断 + 编辑器变更序列分析（本地推理，不依赖API日志）*

| 序号 | 指标名称 | 计算公式 / 采集方式 | 独立开发者洞察（为什么看它？） |
| :--- | :--- | :--- | :--- |
| 25 | **AI代码毛生成率** | AI瞬时插入的行数 / 总代码行数（原始值） | 竞品都在做的基准线。仅凭这个数字无法判断好坏。 |
| 26 | **AI采纳率（即时）** | AI插入后**5分钟内未被撤销/删除**的行数占比 | **第一个核心差异化指标**。AI给的东西你立删，说明它在浪费你时间。 |
| 27 | **AI代码存活率（长期）** | AI生成的代码在**7天后**依然保留在当前文件中的比例 | **终极拷问指标**。AI写的代码能扛住业务变更吗？存活率低说明AI只能写一次性草稿。 |
| 28 | **AI引入错误率** | AI生成后，紧接着出现 `Ctrl+Z` 回退操作的比例 | 反映AI是否经常生成语法错误或逻辑完全跑偏的代码。 |
| 29 | **AI话题漂移度** | 分析短时间内连续AI补全的上下文相似度（本地简单向量化） | **Vibe Drift概念**。高漂移度说明你在被AI带跑偏，从一个需求跳到另一个无关需求。 |
| 30 | **AI依赖疲劳度** | 连续触发AI补全超过5次且每次都要人工修改时，标记为“低效AI时段” | 当你和AI陷入来回拉扯，插件弹窗建议：“**要不关掉AI自己写？可能更快。**” |
| 31 | **AI启动潜伏期** | 从产生空行/停顿到第一次触发AI补全的平均时间 | 快速触发AI = 习惯先问AI；晚触发 = 习惯自己先思考。反映工作流惯性。 |


### 维度七：项目级宏观生态（独立开发者的资产全貌）
*采集层：基于Git仓库根目录、Workspace静态扫描（低频，仅在打开/切换时更新）*

| 序号 | 指标名称 | 计算公式 / 采集方式 | 独立开发者洞察（为什么看它？） |
| :--- | :--- | :--- | :--- |
| 32 | **语言栈占比趋势** | 按周统计项目内各语言（TS/JS/Python/Go等）的净增行数占比 | 独立开发者经常变技术栈。**看这个趋势图，你能发现自己正在逐渐抛弃哪种语言**。 |
| 33 | **项目活跃度温度计** | 近7天内，平均每天有代码变更的项目个数 | 同时活跃项目 > 3个，你的注意力将被严重稀释。 |
| 34 | **仓库体积/依赖膨胀** | 统计 `node_modules` 或 `target` 文件夹大小随时间的变化 | 提醒你哪些项目已经变得臃肿不堪，该做“春清”了。 |
| 35 | **大文件警告** | 统计项目中超过500行的文件数量和占比 | 独立开发者最容易在工具类文件中堆出“万行大泥球”。这是重构的红线信号。 |
| 36 | **遗留代码年龄** | 统计项目中超过**3个月未被编辑**的代码文件数量 | 这些是潜在的“僵尸代码”，如果测试覆盖不到，下个版本可以直接移除。 |
